# รายงานตรวจสอบ Web App — เตรียมส่งอาจารย์ที่ปรึกษา (มิ.ย. 2026)

> ระบบ: OR Flow — บริหารจัดการห้องผ่าตัดศัลยกรรมทั่วไป + ML ทำนายเวลาผ่าตัด
> ขอบเขตการตรวจ: (1) เทคนิค/QA · (2) โครงสร้าง-ฟีเจอร์ · (3) มุมวิจัย
> สรุป: **แอปอยู่ในสภาพพร้อมนำเสนอ** มีจุดที่ควร "ระบุใน limitation" ไม่ใช่บั๊ก

## 1. ผลตรวจเชิงเทคนิค (QA)

| รายการ | ผล |
|---|---|
| Python ทุกไฟล์ compile ผ่าน | ✅ ผ่านทุกไฟล์ |
| โครงสร้าง git | ✅ สะอาด (เหลือ 4 ไฟล์แก้ค้างเล็กน้อย ไม่กระทบ) |
| ความเป็นส่วนตัวผู้ป่วย (PDPA) | ✅ **ไม่มี ชื่อ/HN/AN บน cloud** |
| รองรับ offline | ✅ สลับ `db_mode=sqlite` ใช้ local ได้ |

**PDPA (จุดแข็งที่ควรชูกับอาจารย์):**
- ฐานข้อมูล cloud (Supabase) **ไม่เก็บตัวระบุตัวผู้ป่วยเลย** — ไม่มีคอลัมน์ name / HN / AN
- ใช้ `is_ipd` (0/1) แทนเลข AN — เก็บสถิติ IPD/OPD ได้โดยไม่ระบุตัวคน
- ชื่อผู้ป่วยแสดงบนกระดานเพื่อ identify เท่านั้น (อยู่ใน session ชั่วคราว ไม่ลงฐานข้อมูล)
- โค้ดที่อ้าง AN มี guard รันเฉพาะ local (cloud ข้าม) → ไม่มีทางรั่ว

**ประสิทธิภาพ (ปรับวันนี้):**
- รวม query หน้าสรุปยอด 18→2 · cache สถิติ · reuse connection เดียว (ไม่เปิดใหม่ทุก query)
- หน้าสถิติเลือกโหลดเฉพาะหัวข้อที่ต้องการ → ไม่โหลดทั้งหมดทุกครั้ง

## 2. โครงสร้าง & ฟีเจอร์ (ใช้เป็นโครงคู่มือได้)

### เมนูหลัก (sidebar) — มี password gate ก่อนเข้า
**① 📋 ตารางผ่าตัด** (กระดานติดตามเคสสด)
- อัปโหลด CSV ตารางผ่าตัดประจำวัน · เพิ่มเคส Manual (walk-in/แทรก)
- ติดตามสถานะรายเคส: รอ → เข้าห้อง → ผ่าเสร็จ → จำหน่าย
- 🤖 AI ทำนายเวลาใช้ห้อง · ✏️ พยาบาลแก้เวลาได้ (บันทึก log คน-vs-AI)
- ไฟแดง pulse เคสฉุกเฉิน · Demo mode ลองใช้

**② 📊 บริหารจัดการ** (4 แท็บ)
- **ภาพรวมวันนี้** — KPI สด + ไทม์ไลน์ห้อง
- **สถิติย้อนหลัง** — เลือกช่วงวันที่ + ติ๊กหัวข้อ (KPI / สรุปยอด / กราฟจำนวนเคสรายเดือน / อันดับยอดนิยม / ประสิทธิภาพ / นอกเวลา) → กดแสดงสถิติ + เครื่องมือ import ข้อมูล
- **Utilization** — อัตราใช้ห้อง / turnover / เริ่มทันเวลา / รายห้อง / รายเดือน (ตึกใหม่)
- **AI Prediction (งานวิจัย)** — 2 ส่วน: *คนทั่วไป* (median/กราฟรายวัน/การกระจาย) + *วิทยานิพนธ์* (MAE/R²/scatter/คน-vs-AI)

**③ ⚙️ ตั้งค่า** — ตั้งค่าห้อง/พยาบาล

### เบื้องหลัง (backend)
- ML: XGBoost + target-encoding (median เวลาตามหัตถการ×หมอ) · versioning + champion/challenger
- ฐานข้อมูล: Supabase PostgreSQL (schema `orsurg`) สลับ local SQLite ได้
- 13,402 เคสย้อนหลัง (ปี 64–69) de-identified

## 3. ตรวจมุมวิจัย (สิ่งที่อาจารย์อาจถาม + ที่ควรเขียนใน limitation)

### ✅ จุดที่ทำถูกตามหลักวิจัย
- **กัน data leakage**: ประเมิน AI เฉพาะปี 2568+ (นอกชุดเทรน) — ข้อมูล 64–67 เป็น training set กันออก
- **Utilization** คิดฐานเวลาทำการจริง 8:00–16:00 นับเฉพาะห้อง-วันที่เปิดใช้ (ไม่รวมวันห้องปิด)
- **เลือก elective** ในหน้าประเมิน AI (default) ตรง scope งานวิจัย — toggle รวม emergency ได้
- รายงานหลายตัวชี้วัด: MAE, median, ±15/±30 นาที, R², bias, per-procedure

### ⚠️ ประเด็นที่ควร "ระบุเป็น limitation" ในเล่ม (ไม่ใช่ข้อผิดพลาด)
1. **ความแม่น AI**: MAE เฉลี่ย ~40 นาที (เคสผสม) — อยู่ในกรอบที่งานวิจัย OR-time รายงานกัน · median ~23 นาที (เคสทั่วไป) · แม่นมากในหัตถการที่ทำบ่อย (cystoscopy ±6, tracheostomy ±12) แต่หัตถการเวลาแปรปรวนสูงเดายาก
2. **ชื่อหัตถการกระจาย**: 1,612 ชื่อสำหรับ ~3,000 เคส (HIS ตั้งชื่อไม่เป็นมาตรฐาน) → กระทบความแม่นของหัตถการหายาก
3. **การกันเคสซ้ำ**: ไม่มี identifier (เพื่อ privacy) → ใช้ key composite → หัตถการซ้ำวันเดียวกันอาจยุบ ~5% (undercount เล็กน้อย)
4. **เวลารอผู้ป่วย (wait)**: เก็บได้เฉพาะเคสที่ผ่านกระดาน live (prospective) — ข้อมูลย้อนหลังไม่มี
5. **สถิติย้อนหลัง (หน้า KPI/สรุปยอด)**: *ยังรวมทุก op_type* (ไม่ได้กรอง elective เหมือนหน้า AI) — ถ้า scope วิทยานิพนธ์เป็น elective ล้วน ควรระบุ หรือเพิ่มตัวกรอง
6. **Turnover**: median = 15 นาทีเป๊ะ เพราะ HIS บันทึกเวลาเป็นขั้น 5 นาที (ค่าเกาะกลุ่ม) — ใช้ mean/p90 อธิบายเสริม
