# แผน v2 — รวม OR live + วิจัย (ฉบับจัดกล่องให้ไม่งง)

> อัปเดต 2026-06-07 · ปรับตามเงื่อนไขจริง: บางห้องไม่ใช้บอร์ด, วิจัย=elective, wait มาจากบอร์ด, AI ทำนายรายวันเข้าหน้า AI prediction

---

## ⭐ สรุปสั้นสุด (อ่านอันเดียวพอ)

ข้อมูลทุกอย่างมี **"บ้านเดียว"** — แบ่ง 3 กล่อง ไม่ต้องจำเยอะ:

| กล่อง | ข้อมูล | บ้าน | ที่มา | ครอบคลุม |
|---|---|---|---|---|
| **1. เวลาจริง** | เข้าห้อง/ออกห้อง/ระยะเวลา | `cases` (cloud) | **intraop file (③)** | **ทุกห้อง** (backbone + backup) |
| **2. ของบอร์ด** | **wait_min** + AI ทำนาย + คนแก้ AI | `board_outcome` (cloud, ใหม่) | **live board** ตอนผ่าเสร็จ | เฉพาะห้องที่ใช้บอร์ด |
| **3. ตัวกรอง** | elective เท่านั้น (วิจัย) | ทุก query สถิติ/ML | กรอง `op_type='elective'` | บอร์ดโชว์ครบ, วิจัยกรอง |

> หัวใจ: **③ คือกระดูกสันหลัง** (ครอบคลุมทุกห้อง แม้ห้องที่ไม่ใช้บอร์ด) · **บอร์ดเติมแค่สิ่งที่ ③ ให้ไม่ได้** = เวลารอจริง + AI ทำนาย + คนแก้

---

## 1) ทำไมต้องเก็บทั้ง 2 ทาง (ไม่ใช่งานซ้ำ แต่คนละหน้าที่)

```
บางห้องไม่ใช้บอร์ด ─────────────►  ③ intraop file = backup ครอบคลุมทุกห้อง
                                    (เข้า/ออก/ระยะเวลา/หมอ/หัตถการ → cases)
ห้องที่ใช้บอร์ด ──────────────────►  board เติม: wait_min + AI ทำนาย + คนแก้
                                    (สิ่งเดียวที่ intraop ไม่มี → board_outcome)
```

- **③ ไม่ใช่ "งานซ้ำ"** แต่เป็น **backup หลัก** ของห้องที่ไม่ได้ใช้บอร์ด → ต้องเก็บไว้
- **สิ่งที่ซ้ำจริง** คือเดิมต้อง upload schedule เข้าบอร์ด + ③ → ส่วนนี้ค่อยลดทีหลัง (ดู Phase 2)
- **เวลารอผู้ป่วย (wait_min)** = สิ่งเดียวที่ intraop file ให้ไม่ได้ → ต้องมาจากบอร์ดเท่านั้น

---

## 1.5) ⏱️ เรื่องเวลา — intraop มาช้า ≥ 1 วัน (สำคัญ)

intraop file upload **วันถัดไป** → cases (เวลาทางการ) ตามมา T+1
แต่บอร์ดให้ข้อมูล **วันนั้นทันที** → ยิ่งจำเป็นต้องมี board_outcome

```
วัน T (วันผ่าตัด):   บอร์ด live → board_outcome
                     = wait_min + AI ทำนาย + actual จากบอร์ด (พยาบาลกดเอง)  ✅ มีทันที
วัน T+1:             ③ intraop → cases
                     = เวลาเข้า/ออก/ระยะเวลา "ทางการ" จาก HIS (ทุกห้อง)     ✅ ตามมา
```

**ผลที่ตามมา — เคส elective ที่ใช้บอร์ด จะมี "เวลาจริง 2 ชุด":**
- **บอร์ด (วันนั้น):** actual จากพยาบาลกด — ใช้ดู dashboard/AI เบื้องต้นวันนั้นเลย
- **intraop (วันถัดไป):** actual ทางการจาก HIS — ใช้เป็นความจริงสำหรับสถิติ/ML/งานวิจัย

**กติกาง่ายๆ:**
- **wait_min** → จากบอร์ดเสมอ (intraop ไม่มี) — แหล่งเดียว
- **ระยะเวลาผ่าตัด (duration) สำหรับวิจัย/ML** → ใช้ของ **intraop (ทางการ)** เมื่อมาถึง · ของบอร์ดเป็นตัวพรีวิววันนั้น
- จับคู่ 2 ชุดด้วย (วันที่ + ห้อง + หัตถการ + หมอที่ set) — de-identified
- เพราะเวลาบอร์ด ≠ intraop และมาคนละวัน → **ยิ่งต้องแยก `board_outcome` ออกจาก `cases`** (ไม่ merge ชนกัน)

---

## 2) งานวิจัย = elective เท่านั้น (บอร์ดโชว์ครบ)

- **บอร์ด:** แสดงทุกเคส (elective / emergency / urgent / นอกเวลา) — พยาบาลใช้งานครบ
- **สถิติ + ML + AI prediction:** กรอง `op_type = 'elective'` เท่านั้น
- เก็บ `op_type` ทุกเคสไว้ → ใช้เป็นตัวกรอง (เก็บครบ, กรองตอนวิเคราะห์)

---

## 3) หน้า AI Prediction — เอา AI ทำนายรายวันมาประเมิน

- ทุกเคสที่บอร์ดทำนาย → เก็บ `ai_predicted_min` (วันที่ทำนาย) ลง `board_outcome`
- พอรู้เวลาจริง (ผ่าเสร็จ / ③) → เทียบ **AI ทำนาย vs จริง** รายวัน
- หน้า AI prediction โชว์: MAE รายวัน/รายเดือน, แนวโน้ม, เฉพาะ elective, เฉพาะปี 68+ (กันชุดเทรน — มีอยู่แล้ว)
- **คน vs AI** (override_log) = อีกมุมหนึ่ง: เคสที่คนแก้ AI แม่นกว่าไหม

---

## 4) สิ่งที่ต้องทำ (เล็กกว่าแผน v1)

### Phase 1 — เก็บข้อมูลบอร์ดขึ้น cloud (de-identified)
1. **table ใหม่ `board_outcome`** (cloud, ไม่มี name/hn/an):
   `op_date, op_type, is_ipd, procedure_name, surgeon_name, division_code, room_no, age, ai_predicted_min, user_override_min, actual_duration_min, wait_min, arrived_at, in_or_at, op_end_at, source='board'`
   → เป็น cloud version ของ `case_history.csv` ที่มีอยู่แล้ว
2. **ฟังก์ชัน `save_board_outcome(case)`** + ต่อเข้า `_do_finish` (ข้าม demo)
3. **หน้า AI prediction:** ดึง predicted vs actual จาก `board_outcome` + `cases` (กรอง elective + ปี 68+)
4. **หน้าสถิติ/ML:** เพิ่มกรอง `op_type='elective'`
5. **③ คงไว้เป็น backbone** — ปรับคำอธิบายว่าเป็น "backup ครอบคลุมทุกห้อง + ย้อนหลัง"

### Phase 2 — ลด upload ซ้ำ (อนาคต)
- ให้ upload schedule บนบอร์ดเขียน "scheduled" ขึ้น cloud ด้วย → ③ เหลือไว้เฉพาะวันที่ไม่ใช้บอร์ดจริงๆ

---

## 5) Privacy + Edge cases

- `board_outcome` ไม่มี name/hn/an · ผูกเคสด้วยข้อมูล de-identified เท่านั้น ✅
- **ข้าม demo:** ไม่เขียนเคสทดลอง
- **บอร์ด vs intraop เวลาเข้าห้องไม่ตรงกัน** → จึงแยก `board_outcome` ออกจาก `cases` (ไม่ merge เวลาชนกัน) · วิเคราะห์ค่อย join ตาม (วันที่+ห้อง+หัตถการ)
- **เคส emergency/urgent** → บอร์ดเก็บ wait ได้ปกติ แต่ตอนวิจัยกรองออกด้วย op_type
- `case_history.csv` (local) → เก็บต่อเป็น backup ในเครื่องได้ (board_outcome = ตัวเดียวกันบน cloud)

---

## 6) จุดตัดสินใจที่เหลือ (น้อยลงแล้ว)

1. **`board_outcome` แยก table** (แนะนำ) หรือยัดเข้า `cases`? → แยกดีกว่า กันเวลา board≠intraop ชนกัน
2. **wait_min ของเคส emergency** → เก็บไว้ดูด้วยไหม (เก็บได้ กรอง elective ตอนวิจัย)
3. เริ่ม **Phase 1** เลยไหม หรือปรับแผนเพิ่ม

---

### สรุปให้ไม่งง
- **③ = backbone** (ทุกห้อง, เวลาจริง, backup) — เก็บไว้
- **บอร์ด = เติม wait + AI ทำนาย + คนแก้** ลง `board_outcome` (cloud, de-identified)
- **วิจัย = elective** (กรอง op_type)
- **หน้า AI prediction** = เทียบ AI ทำนาย vs จริง รายวัน (elective, ปี 68+)
- ทำ **Phase 1** ก่อน = table เดียว + ฟังก์ชันเดียว + ต่อเข้าผ่าเสร็จ → ไม่กระทบ flow พยาบาล
