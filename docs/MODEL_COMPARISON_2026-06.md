# เปรียบเทียบโมเดลทำนายเวลาผ่าตัด — หลักฐานการเลือกโมเดล (สำหรับสอบป้องกัน)
*รันจริง 10 มิ.ย. 2026 · สคริปต์ `compare_models.py` (seed=42, ทำซ้ำได้) · ขอบเขตข้อมูลตาม ethics: พ.ศ. 2564–2567 เท่านั้น*

## 1. ระเบียบวิธี

- **Matched cohort — ทุกตาราง n เท่ากัน:** ใช้เฉพาะเคสที่ข้อมูลเวลา "ครบและสมเหตุสมผล
  ทั้งสองนิยาม" (room-in ≤ ลงมีด ≤ ปิดแผล ≤ room-out และ duration 5–1,440 นาที)
  → คัดออก: เทรน 67/5,650 (1.2%) · ทดสอบ 12/2,004 (0.6%)
  → **n_train = 5,583 · n_test = 1,992 เท่ากันทั้งสองตาราง**
  *(ประโยคใน methodology: "วิเคราะห์บนเคสที่บันทึกเวลาครบถ้วนและสอดคล้องทุกจุดเวลา
  ร้อยละ 98.8 ของเคสทั้งหมด")*
- **แบ่งข้อมูลตามเวลา (temporal split):** เทรน พ.ศ. 2564–2566 → ทดสอบ พ.ศ. 2567
- **median/encoding ทุกตัวคำนวณจากชุดเทรนเท่านั้น** — ไม่มี data leakage
- **ทุกโมเดล ML (ข้อ 4–7) เห็น feature ชุดเดียวกัน** (รวมค่า hier เป็น feature ด้วย)
  → ความต่างที่เหลือ = อัลกอริทึม/สถาปัตยกรรมล้วนๆ — เทียบแฟร์
- XGBoost ใช้ early stopping ด้วยปีท้ายของชุดเทรน (2566) ตามสเปคโมเดลจริงในเล่ม

### การตีความเมตริก (สำคัญตอนตอบกรรมการ)

- **MAE = ตัวชี้วัดหลัก** ตามธรรมเนียมงาน OR duration prediction (งาน XGBoost/RF
  ด้านห้องผ่าตัดส่วนใหญ่รายงาน MAE เป็นหลัก) — โมเดลที่เลือกเทรนด้วย MAE objective โดยตรง
- **RMSE / R² ไวต่อเคสสุดโต่ง (outlier):** Random Forest เฉลี่ยจากหลายต้นไม้ → ทำนาย
  "เรียบกว่า" กับเคสหางยาว จึงชนะ RMSE/R² เล็กน้อย (R² 0.542 vs 0.475) **แลกกับ**
  สัดส่วนทำนายแม่น ±15/±30 นาทีที่แย่กว่าชัดเจน (31.0% vs 33.7% · 53.8% vs 56.6%)
  → ถ้าเป้าหมายคือ "จัดตารางรายเคสให้คลาดน้อย" เกณฑ์ ±15/±30 ตรงกว่า RMSE
- **R² ของ Naive ติดลบ** เป็นเรื่องปกติ: ตัวทำนายคงที่ที่ใช้ "มัธยฐาน" (ไม่ใช่ mean)
  จะได้ R² ≤ 0 โดยนิยาม — ใช้เป็น baseline อ้างอิงเท่านั้น
- **MAPE สูงใน "เวลาผ่าตัดสุทธิ" (61.8%) เป็นธรรมชาติของข้อมูล** ไม่ใช่ความผิดของโมเดล:
  เคสสั้นมาก (เช่น 10–15 นาที) พลาดเพียง 8–10 นาทีก็คิดเป็น 60–80% แล้ว — วรรณกรรม
  จึงให้น้ำหนัก MAPE ต่ำสุดในสี่ตัว · เวลารายงานควรกำกับเสมอว่า median ของเวลาจริง
  คือ ~65 นาที (เคสสั้นเยอะ)

## 4. การอ่านผล — 4 ข้อค้นพบ

**(1) "ความรู้โครงสร้างหัตถการ" สำคัญกว่าตัวอัลกอริทึม**
การทดลองเสริม: ถ้าให้ RF/XGBoost ทำนายตรงโดย*ไม่มี*ค่า hier เป็น feature
(มีแค่ median แยกชั้นแบบไม่มีเกณฑ์ min_count) MAE จะแย่ลงเหลือ ~54 นาที
— ใส่ hier เข้าไป ทุกโมเดลดีขึ้นมาเกาะกลุ่ม ~42 ทันที
→ สิ่งที่ขับความแม่นคือ "มัธยฐานลำดับชั้นที่มีเกณฑ์ข้อมูลขั้นต่ำ" ไม่ใช่ความซับซ้อนของ ML

**(2) เหตุผลที่ข้อมูลนี้ต้องมี hier: long-tail รุนแรง**
ในชุดทดสอบ: **48.3% ของเคส** (= 74% ของรายชื่อหัตถการ) มีชื่อเต็มที่*ไม่เคยปรากฏ*
ในชุดเทรน → โมเดลที่ผูกกับชื่อเต็มอย่างเดียวพังครึ่งหนึ่งของงาน
แต่ถอยไประดับ keyword คำแรก เหลือเคสที่ไม่รู้จักเพียง 5.6% — นี่คือกลไกของ hier
*(หมายเหตุ: เล่มปัจจุบันเขียน "59%" — ควรระบุนิยามให้ชัดว่านับ % ของเคส (48.3%)
หรือ % ของรายชื่อ (74%) แล้วแก้เล่มให้ตรงนิยามเดียว)*

**(3) RF กับ XGBoost "เสมอกัน" ทาง MAE — ตัดสินด้วยเกณฑ์อื่น**
ความต่าง MAE ~0.5 นาที ไม่มีนัยสำคัญ (CI ครอบ 0) การเลือก hier+XGB residual จึงอิง:
- **สัดส่วนทำนายแม่น ±15/±30 นาทีดีกว่า RF ชัดเจน** (33.7% vs 31.0% · 56.6% vs 53.8%)
  — เกณฑ์นี้มีความหมายตรงต่อการจัดตารางห้องมากกว่า MAE เฉลี่ย
- **อธิบายได้:** ฐานทำนาย = "มัธยฐานของเคสคล้ายกัน n เคส ที่ระดับชั้นใด" → แอพแสดง
  หลักฐานให้พยาบาลตัดสินใจเชื่อ/แก้ค่าได้ (human override) — RF/XGB ทำนายตรงให้แบบนี้ไม่ได้
- **ปลอดภัยเชิงระบบ:** ถ้าชั้น ML ล้มเหลว ระบบถอยไปใช้ hier (ค่ามัธยฐานจริง) ได้ทันที
- **ขนาด/ความเร็ว:** XGB residual (depth 3) ~0.9 MB เหมาะกับ deploy บน cloud ฟรี —
  RF 400 ต้นใหญ่กว่าหลายสิบเท่า
- RF ได้ RMSE ดีกว่าเล็กน้อย (เรียบกับเคสสุดโต่ง) — เป็น trade-off ที่ยอมรับ เพราะเคสหางยาว
  จัดการด้วยกลไก alert เกินเวลา + คนแก้ค่า ไม่ใช่หน้าที่ของ point prediction

**(4) ทำไมต้อง temporal split**
แบ่งแบบ random (ผิดหลักการกับข้อมูลเวลา) ให้ MAE 38.9 — ดูดีกว่าจริง ~3.5 นาที
เพราะเคสร่วมยุค/หัตถการชุดเดียวกันรั่วข้าม train↔test
→ งานนี้รายงานแบบ temporal ซึ่ง "เลขแย่กว่าแต่ซื่อสัตย์" และตรงกับการใช้จริง (ทำนายอนาคตจากอดีต)

## 5. แนวคำตอบกรรมการ (พูดได้เลย)

**ถาม: ทำไมไม่ใช้ Random Forest?**
ตอบ: ได้ทดลองแล้วบน locked test set เดียวกัน feature ชุดเดียวกัน — MAE ของ RF (42.2)
กับโมเดลที่เลือก (42.7) ไม่ต่างกันอย่างมีนัยสำคัญ (paired bootstrap 95% CI [−0.38, +1.48])
แต่โมเดลที่เลือกชนะในสัดส่วนทำนายคลาดไม่เกิน 15/30 นาที (33.7% vs 31.0% · 56.6% vs 53.8%)
ซึ่งเป็นเกณฑ์ที่มีความหมายต่อการจัดตารางห้องผ่าตัด และให้ความสามารถอธิบายผ่านฐาน
มัธยฐานลำดับชั้น ซึ่งจำเป็นต่อการยอมรับของผู้ใช้ทางคลินิก

**ถาม: ทำไมไม่ใช้ XGBoost เดี่ยวๆ?**
ตอบ: เชิงตัวเลขเท่ากัน (42.4 เท่ากัน) แต่สถาปัตยกรรม hier+residual ให้สองสิ่งที่ XGBoost
เดี่ยวไม่ให้: (1) คำอธิบาย "อิงจากเคสคล้ายกี่เคส ระดับความเฉพาะใด" ที่แสดงต่อพยาบาล
(2) fallback ปลอดภัย — ถ้าชั้น ML ใช้ไม่ได้ ระบบยังทำนายด้วยมัธยฐานจริงของเคสคล้ายได้

**ถาม: แล้ว deep learning / LSTM?**
ตอบ: ข้อมูล tabular ขนาด ~6 พันเคสต่อชุดเทรน พร้อม long-tail รุนแรง — วรรณกรรม
และผลทดลองชี้ตรงกันว่า gradient boosting/median-based เหนือกว่าบนขนาดนี้
neural network ต้องการข้อมูลมากกว่าและอธิบายยากกว่า ไม่สอดคล้องบริบทการนำไปใช้

**ถาม: ทำไมไม่แบ่งข้อมูลแบบ 80/20?**
ตอบ: สัดส่วนจริงใกล้เคียง 80/20 อยู่แล้ว (เทรน 5,583 / ทดสอบ 1,992 ≈ 74/26) —
สิ่งที่ต่างคือ "วิธีเลือก": ไม่ใช้การสุ่ม แต่แบ่งตามเวลา (เทรนปี 2564–2566 → ทดสอบปี 2567
ทั้งปี) เพราะข้อมูลมีมิติเวลาและงานจริงของระบบคือทำนายเคสในอนาคตจากข้อมูลอดีต
หากสุ่ม เคสร่วมยุค/หัตถการเดียวกันจะกระจายอยู่ทั้งสองฝั่ง และ median encoding จะถูก
คำนวณจากเคสร่วมช่วงเวลาเดียวกับชุดทดสอบ เกิด temporal leakage — ได้สาธิตด้วย
โมเดลตัวเดียวกัน: แบ่ง random ให้ MAE 39.3 เทียบกับ temporal 42.7 (ดูดีเกินจริง
~3.4 นาที) จึงเลือกรายงานแบบ temporal ตามแนวทาง TRIPOD ซึ่งจัด temporal validation
เป็นหลักฐานที่แข็งแรงกว่า random internal validation
(ถ้าถามต่อเรื่อง k-fold: k-fold แบบสุ่มผิดด้วยเหตุผลเดียวกัน — ทางที่ถูกคือ
rolling-origin CV ซึ่งเสนอเป็นงานต่อยอดได้)

**ถาม: ตัวเลขเชื่อได้แค่ไหน?**
ตอบ: ประเมินแบบ out-of-sample ตามเวลา (เทรนอดีต→ทดสอบอนาคต) encoding จากชุดเทรน
เท่านั้น ทำซ้ำได้ด้วย seed คงที่ และเมื่อสาธิตแบ่งข้อมูลแบบ random จะได้ตัวเลขดีเกินจริง
~3.5 นาที — เป็นเหตุผลที่งานนี้เลือกรายงานแบบ temporal
