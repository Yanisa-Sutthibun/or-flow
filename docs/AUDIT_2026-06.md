## 👨‍💻 มุมวิศวกรรมคอมพิวเตอร์ / QA

### 🔴 CRITICAL-1 — `_get_table_columns` ใช้ schema ผิด (แก้แล้ว ✅)
`import_historical.py:36` query `information_schema` ด้วย `table_schema='public'` แต่แอปใช้ schema `orsurg` → ในโหมด supabase คืน **set ว่าง** → ทุก `if 'col' not in existing_cols` ติด True → สั่ง `ALTER TABLE ADD COLUMN` ซ้ำ → รอบสองคอลัมน์มีอยู่แล้ว → psycopg2 error → **transaction abort → import พังทั้งยวง**
- **แก้แล้ว:** เปลี่ยนเป็น `current_schema()` (เหมือนที่ `main_or_db._table_columns` ทำถูกอยู่แล้ว) + เพิ่ม `conn.rollback()` ใน except ของ 2 ALTER ในพาธ live (กัน transaction ค้าง)
- หมายเหตุดี: **ปุ่ม ③ ที่ใช้อยู่ปลอดภัย** — `reimport_timestamps` ไม่มี ALTER เลย (re-run ③ ได้) · จุดเสี่ยงคือ auto-import (`import_historical`) ซึ่งแก้แล้ว

### 🔵 LOW-1 — `add_walkin_case` คืน `lastrowid` = None บน Postgres
`main_or_db.py` ใช้ `cur.lastrowid` หลัง INSERT — wrapper Postgres คืน None (ไม่มี RETURNING) · **ตอนนี้ไม่กระทบ** เพราะ caller เดียว (main_or_tracking, dead page) ไม่ใช้ค่า return · เป็น landmine ถ้ามีคนเรียกใช้ id ในอนาคต → แนะนำเติม `RETURNING case_id`

### 🔵 LOW-2 — โค้ดตายยังอ้างคอลัมน์ name/hn/an
`import_historical.py` มี INSERT ที่ hardcode `(op_date, hn, name, ...)` 2 จุด (บรรทัด ~855, ~1196) ในฟังก์ชัน `import_walkins_from_cost_excel` / `import_cost_driven` — **ไม่ถูกเรียกใช้** (dead) และถ้าเรียกจะ error (คอลัมน์ไม่มีบน cloud) = fail-safe ไม่ leak · แต่ควรลบ/แก้กันพลาดในอนาคต

### 🔵 LOW-3 — สคริปต์ supabase ล้าสมัย
`supabase/anonymize_for_cloud.py` และ `inspect_db.py` ยังอ้าง name/hn/an (schema เดิม) → จะ error ถ้ารัน · เป็น standalone ไม่กระทบแอป · แนะนำลบ/อัปเดต

### 🔵 LOW-4 — `secrets.toml.example` มี app_password default
ไฟล์ตัวอย่าง (tracked ใน git) มี `app_password = "minor_or_2569"` — ไม่ใช่ความลับ DB และไม่ใช่ PII ผู้ป่วย แต่ควรเตือนให้เปลี่ยนตอน production

### ✅ ตรวจแล้วปลอดภัย (Postgres compat)
ตัวแปลง SQL จัดการครบ: `strftime`, `DATE('now')`, `datetime('now')`, `AUTOINCREMENT→SERIAL`, `PRAGMA`, `INSERT OR IGNORE→ON CONFLICT`, escape %, `?→%s` · `ROUND/AVG/SUM`, `GROUP BY alias`, `LIKE` (lowercase ทั้งคู่), `_set_app_setting`/`save_room_settings` ใช้ `ON CONFLICT DO UPDATE` แล้ว · `init_db` return ก่อนสำหรับ PG (executescript ไม่รันบน cloud) · `pd.read_sql_query` ผ่าน wrapper ทำงานได้

## 🔬 มุมนักวิจัยบริหารการพยาบาล (คุณภาพข้อมูล)

### 🟣 RESEARCH-1 — ยังไม่มี filter "elective เท่านั้น" ในสถิติ/ML
ตามที่คุยกันว่างานวิจัยทำเฉพาะ elective แต่ตอนนี้ query สถิติ/ML/AI prediction **รวมเคส emergency ด้วย** (เห็น op_type='elective' เป็นแค่ default/ตัวนับย่อย ไม่ใช่ตัวกรองหลัก) · ปี 69 ที่เพิ่ง import มี emergency ~235 เคสปนอยู่
- **ผลต่อวิจัย:** ตัวเลข MAE/utilization/สถิติ ปนเคสฉุกเฉิน → ไม่ตรง scope วิทยานิพนธ์
- แนะนำ: เพิ่มตัวกรอง `op_type='elective'` (หรือ toggle เลือกได้) ในหน้าสถิติ/AI prediction/ML training — อยู่ในแผน board_outcome แล้วแต่ยังไม่ลงมือ

### 🟣 RESEARCH-2 — การกันซ้ำแบบไม่มี identifier ทำเคสซ้ำหาย ~5%
เพราะ DB ไม่มี an/hn การกันซ้ำใช้ (วันที่+ห้อง+หัตถการ+หมอ/เวลาเข้าห้อง) → **หัตถการเดียวกัน หมอเดิม ห้องเดิม วันเดียวกัน หลายเคส → ยุบเป็นเคสเดียว** · ปี 69: รวม 1,672 → import จริง 1,589 (หาย ~83, ~5%)
- **ผลต่อวิจัย:** undercount ปริมาณงาน + เสีย training sample ของหัตถการที่ทำซ้ำบ่อย (เช่น EC, cystoscopy)
- แนะนำ: ถ้าต้องการ count แม่น เพิ่ม field "เวลานัด/ลำดับเคส" (ไม่ใช่ PII) เป็นตัวแยก — เป็น trade-off ที่ยอมรับได้ถ้าเน้น privacy แต่ควรระบุใน limitation ของวิทยานิพนธ์

### 🟣 RESEARCH-3 — wait_min ไม่มีในข้อมูล historical
ไฟล์ intraop (`_parse_intraop`) ไม่มีเวลา "ผู้ป่วยมาถึง" → เคส import (③, migrate) มี `wait_min` ว่าง/0 · wait มีเฉพาะเคสที่บันทึกผ่านบอร์ด live (แผน board_outcome)
- **ผลต่อวิจัย:** สถิติ "เวลารอผู้ป่วย" ใช้ได้เฉพาะช่วงที่ใช้บอร์ด live เท่านั้น ไม่ใช่ย้อนหลัง
- แนะนำ: ระบุชัดในวิจัยว่า wait time เก็บ prospective (จากบอร์ด) · หรือถ้า intraop มีคอลัมน์เวลามาถึง ให้ดึงเพิ่ม
