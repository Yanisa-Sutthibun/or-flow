# Flow การทำงานของ OR Flow (Web App) — ฉบับอัปเดตหลัง hardening

ระบบจัดการคิวห้องผ่าตัด + AI ทำนายเวลา · Streamlit + Supabase (PostgreSQL) · ใช้พร้อมกันได้ 10+ เครื่อง

---

## 1) ภาพรวม + บทบาทผู้ใช้

ทุกเครื่องเปิดเว็บเดียวกัน เห็น **บอร์ดกลางชุดเดียวกัน** (ซิงก์ผ่าน Supabase)

| บทบาท | ทำอะไร | หน้าที่ใช้ |
|---|---|---|
| พยาบาลหน้างาน | กดสถานะเคส (รับเข้า/เข้าห้อง/ผ่าเสร็จ/จำหน่าย) · แก้เวลา ✏️ | ตารางผ่าตัด (OR Board) |
| ผู้บริหาร/หัวหน้า | ดู flow สด ทุกห้อง · KPI · ความแม่น AI | หน้าผู้บริหาร |
| ผู้ดูแล (Mukky) | อัปโหลดตาราง · นำเข้าย้อนหลัง · rebackfill · ลบข้อมูล | Maintenance (ล็อกด้วย PIN) |

---

## 2) Flow หลัก — วงจรชีวิต 1 เคส (5 สถานะ)

```
⬜ ยังไม่มา  →  🟡 รอผ่าตัด  →  🔵 กำลังผ่า  →  🚪 รอจำหน่าย  →  ✅ จำหน่ายแล้ว
   [รับเข้า]      [เข้าห้อง]       [ผ่าเสร็จ]       [จำหน่าย]
```

แต่ละขั้น พยาบาลกดปุ่มหลักของแถวนั้น · มีปุ่ม **↩ ย้อนกลับ** (กันกดพลาด) และ **✏️ แก้เวลา** ตลอดแถว

- **รับเข้า (มาถึง holding):** เริ่มจับ "เวลารอ" + นาฬิกานับเดินหน้า
- **เข้าห้อง:** บันทึกเวลาเข้าห้อง · กันเข้าห้องซ้ำ (ห้องที่มีเคสกำลังผ่าอยู่ กดไม่ได้)
- **ผ่าเสร็จ:** บันทึกเวลาออกห้อง → คำนวณเวลาครองห้องจริง · เก็บเข้า override_log (เทียบ คน-vs-AI)
- **จำหน่าย:** ปิดเคส

> เคสที่ผู้ป่วยมาแล้วแต่ยังไม่ได้ผ่าเกิน 15:30 น. จะถูกดึงขึ้น section "รับเวร" อัตโนมัติ

---

## 3) บอร์ดกลางหลายเครื่อง (หัวใจของระบบ multi-user)

ทุกเครื่องแชร์บอร์ดเดียวกันผ่าน Supabase โดย:

1. **ดึงอัตโนมัติทุก ~30 วินาที** (auto-refresh) + ปุ่ม **🔄 รีเฟรช** ดึงทันที
2. **เซฟเฉพาะตอน "เครื่องนี้กดเปลี่ยนสถานะจริง"** (dirty flag) — เปิด popover/รีเฟรชเฉยๆ ไม่เขียนทับใคร
3. **กันงานหาย (optimistic merge):** ก่อนเขียน เช็ก version ใน DB ถ้าเครื่องอื่นเขียนแซง → merge เฉพาะเคสที่เครื่องนี้แก้ลงบนของล่าสุด (ไม่ทับทั้งกระดาน)
4. **เซฟล้มเหลว → คง dirty + retry + เตือน** "⚠️ บอร์ดกลางออฟไลน์ — เครื่องนี้ยังไม่แชร์" (ไม่โกหกว่าซิงก์แล้ว)
5. **ข้ามเที่ยงคืน** → ล้างเคสเมื่อวานอัตโนมัติ ดึงบอร์ดวันใหม่
6. **ปุ่มล้างกระดานวันนี้** (สำหรับลบเคสทดสอบ) → เขียน payload ว่าง + version+1 ให้ทุกเครื่องว่างตาม (ไม่ใช่ลบ key ที่ทำให้เคสผีคืนชีพ)

ภาระ DB: 12 เครื่อง ≈ 0.6 connection พร้อมกันโดยเฉลี่ย จาก pool 20 (retry+backoff ไม่มี closeall ตัดทุกคน) — รองรับ burst เช้าได้

---

## 4) AI ทำนายเวลา + การแก้เวลา (✏️)

- โมเดล **honest_v1** (hierarchical median + XGBoost residual) ทำนาย **เวลาครองห้องผ่าตัด** จาก: หัตถการ, แพทย์, สาขา, อายุ, ชั่วโมงนัด, วันในสัปดาห์, เดือน, ห้อง
- แสดง **ช่วงทำนาย (conformal)** เช่น 100 ± 103 นาที ที่ความเชื่อมั่น 90% (การันตีจาก hold-out ปี 2567)
- พยาบาลกด **✏️** แก้เวลาเองได้ (human override ชนะ AI) → ระบบเก็บทั้งค่า AI และค่าที่คนแก้ลง override_log เพื่อเทียบความแม่น
- โมเดลเทรนถึง พ.ศ. 2566 · ทดสอบ 2567 · MAE ≈ 41.8 นาที (เวลาครองห้อง)

## 5) การนำเข้าข้อมูล

- **อัปโหลด CSV ตาราง (HIS)** → ทำนายเวลาให้ทุกเคส แล้วขึ้นบอร์ด (ล็อก PIN ผู้ดูแล)
- **เพิ่มเคสเอง (Walk-in/แทรก)** → กรอกฟอร์ม AI ทำนายให้ทันที
- **id เคสเป็น deterministic hash** (hn·ชื่อ·วันที่·หัตถการ·เวลา·ลำดับ) → ถ้า 2 เครื่องอัปโหลดตารางเดียวกัน ไม่เกิดเคสซ้ำ

---

## 6) ความเป็นส่วนตัว (PDPA · มาตรา 3.6.4)

- ชื่อผู้ป่วยบนบอร์ด/บนทุกที่ที่ขึ้น cloud แสดงแบบ **ย่อเสมอ**: คำนำหน้า/ยศ + ชื่อต้น + นามสกุลย่อ + HN 4 ตัวท้าย (เช่น `น.ส. ญาณิศา ส. · HN …9567`) — **ไม่มีโหมดชื่อเต็ม** ไม่แสดงอายุ
- ชื่อแพทย์/พยาบาลในไฟล์โมเดล + DB ถูกแปลงเป็นรหัส `SURG_xxx` (รวม override_log)
- `procnote` (free text) ไม่ขึ้น cloud · ทุกค่าที่ฝัง HTML ผ่าน `html.escape` (กัน XSS)
- บอร์ดกลางบน cloud **ลบ snapshot อัตโนมัติเมื่อเก่ากว่า 7 วัน** (data retention)

---

## 7) Maintenance (ผู้ดูแล · ล็อก PIN)

- **นำเข้าย้อนหลัง** 64–67 · **Rebackfill AI** (คำนวณค่าทำนายใหม่ + บันทึก source จริงลง ai_model_ver)
- **ลบข้อมูล:** ลบทั้งหมด (ต้องพิมพ์ "DELETE" ยืนยัน + สำรองอัตโนมัติก่อนลบ) หรือ ลบเฉพาะช่วงวันที่
- หน้า "ความแม่น AI" คิดจากเคสจริงตั้งแต่ 2025 เป็นต้นไป เฉพาะที่โมเดลจริงทำนาย (ตัด fallback ออก)

---

## 8) Data Flow (ภาพรวมการไหลของข้อมูล)

```
HIS (CSV ตาราง)
      │  อัปโหลด + ทำนายเวลา (honest_v1)
      ▼
  เครื่องพยาบาล  ──กดสถานะ──►  เซฟ (mask ชื่อ/HN) ─► Supabase (บอร์ดกลาง + cases)
      ▲                                                      │
      └──────────── ดึงทุก 30 วิ / merge ◄───────────────────┘
                                                             │
                              ผู้บริหาร ◄── KPI/flow สด ───────┘
```

หลักการ: **ข้อมูลระบุตัวตนถูก mask ก่อนออกจากเครื่องเสมอ** · ทุกเครื่องเห็นสถานะตรงกันใน ~30 วิ · งานไม่หายแม้หลายคนกดพร้อมกัน
