# โปรโตคอลทดสอบ Real-time ข้ามเครื่อง + การบันทึกบน Supabase

วัตถุประสงค์: พิสูจน์ว่าระบบ (1) ซิงก์ข้ามเครื่องแบบ real-time (2) บันทึกข้อมูลบน Supabase จริง (3) งานไม่หายเมื่อหลายคนแก้พร้อมกัน — เพื่อแนบเป็นหลักฐานประกอบวิทยานิพนธ์

## สิ่งที่ต้องเตรียม
- เครื่อง A และเครื่อง B (คนละเครื่อง/คนละเน็ตยิ่งดี) เปิดแอปทั้งคู่ `streamlit run main_or_app.py`
- ตั้ง `db_mode = "supabase"` ใน secrets ทั้ง 2 เครื่อง (ทดสอบบนเซิร์ฟเวอร์จริง)
- เปิดหน้า "ตารางผ่าตัด" วันเดียวกันทั้ง 2 เครื่อง · เตรียมนาฬิกา/มือถือจับเวลาไว้แคปหน้าจอ
- โหลดเคสทดสอบ (ชื่อปลอม) เข้าบอร์ด 3–5 เคส

> หลักฐานที่ "ปลอมไม่ได้" = แคปหน้าจอ 2 เครื่องคู่กัน (มีนาฬิกาในจอ) + ผลรัน `python verify_supabase.py`

---

## ชุดทดสอบ (ทำทีละข้อ บันทึกผลในตารางท้ายเอกสาร)

**T1 — ซิงก์ A→B:** เครื่อง A กด "เข้าห้อง" เคสหนึ่ง
- คาดหวัง: เครื่อง B เห็นเคสนั้นเป็น "กำลังผ่า" ภายใน ~30 วิ (หรือกด 🔄 เห็นทันที)
- หลักฐาน: แคป A (เวลา T) + แคป B (เวลา T+30 วิ)

**T2 — ซิงก์ B→A:** เครื่อง B กด ✏️ แก้เวลาคาดการณ์เคสหนึ่ง
- คาดหวัง: เครื่อง A เห็นค่าเวลาใหม่ภายใน ~30 วิ
- หลักฐาน: แคป B + แคป A

**T3 — แก้พร้อมกันไม่ทับกัน (lost-update):** A แก้เคส #1 และ B แก้เคส #2 "ในเวลาไล่เลี่ยกัน"
- คาดหวัง: หลังซิงก์ ทั้งเคส #1 และ #2 อัปเดตครบทั้ง 2 เครื่อง (ไม่มีอันไหนหาย)
- หลักฐาน: แคปทั้ง 2 เครื่องหลังซิงก์

**T4 — บันทึกบน Supabase จริง (persistence):**
- รัน `python verify_supabase.py` บนเครื่อง A → จดค่า `version` และ `saved_at`
- รัน `python verify_supabase.py` บนเครื่อง B → **ค่าต้องตรงกัน** = ข้อมูลมาจากเซิร์ฟเวอร์กลางชุดเดียวกัน
- (เสริม) เปิด Supabase Dashboard → SQL Editor → รัน:
  ```sql
  SELECT key, left(value, 80) AS preview
  FROM orsurg.app_settings
  WHERE key LIKE 'board_state_%';
  ```
  → เห็นแถว `board_state_<วันนี้>` = ยืนยันว่าข้อมูลอยู่บนเซิร์ฟเวอร์
- หลักฐาน: ผลรันสคริปต์ทั้ง 2 เครื่อง + screenshot SQL Editor

**T5 — ข้อมูลไม่หายหลังปิด-เปิด (proves server-side):** ปิดแอปเครื่อง A แล้วเปิดใหม่
- คาดหวัง: บอร์ดเด้งกลับมาเหมือนเดิม (ดึงจากเซิร์ฟเวอร์ ไม่ใช่ไฟล์ในเครื่อง)
- หลักฐาน: แคปก่อนปิด + หลังเปิดใหม่

**T6 — แจ้งเตือนเมื่อออฟไลน์ (honest failure):** ตัดเน็ตเครื่อง A ชั่วคราว แล้วกดปุ่ม
- คาดหวัง: ขึ้น ⚠️ "บอร์ดกลางออฟไลน์ — เครื่องนี้ยังไม่แชร์" (ไม่หลอกว่าซิงก์แล้ว)
- หลักฐาน: แคปข้อความเตือน

**T7 — ล้างกระดานข้ามเครื่อง:** เครื่อง A กด "ล้างกระดานวันนี้"
- คาดหวัง: เครื่อง B บอร์ดว่างภายใน ~30 วิ
- หลักฐาน: แคปทั้ง 2 เครื่อง

## ตารางบันทึกผล (กรอกแล้วแนบส่งอาจารย์)

| ข้อ | สิ่งที่ทดสอบ | เวลา A | เวลา B เห็น | ผ่าน/ไม่ผ่าน | ไฟล์หลักฐาน |
|---|---|---|---|---|---|
| T1 | ซิงก์ A→B (เข้าห้อง) | | | | |
| T2 | ซิงก์ B→A (แก้เวลา ✏️) | | | | |
| T3 | แก้พร้อมกันไม่ทับ | | | | |
| T4 | version/saved_at ตรงกัน 2 เครื่อง | | | | |
| T5 | ปิด-เปิด ข้อมูลคงอยู่ | | | | |
| T6 | แจ้งเตือนออฟไลน์ | | | | |
| T7 | ล้างกระดานข้ามเครื่อง | | | | |

**สรุป:** ระบบบันทึกข้อมูลบนเซิร์ฟเวอร์กลาง (Supabase) และซิงก์ข้ามเครื่องภายใน ~30 วินาที
โดยข้อมูลผู้ป่วยถูกปกปิด (mask) ก่อนขึ้น cloud · งานไม่สูญหายเมื่อใช้พร้อมกันหลายเครื่อง

> หมายเหตุข้อจำกัด (เขียนในบทที่ 5): การ merge ใช้ optimistic concurrency ถ้า 2 เครื่อง
> เซฟ "เคสเดียวกัน" ห่างกัน < ~300ms ยังมีโอกาสทับกันระดับเคส (พบยากมาก ผลจำกัด)
